# План оптимизации производительности TSV Converter v11.0

Данный документ описывает стратегию оптимизации приложения для повышения скорости конвертации, отзывчивости интерфейса (UI) и эффективности использования памяти.

---

## 1. Устранение двойного чтения файлов (High Impact)

**Проблема:**
В текущей реализации (`converter.py`, метод `run`) происходит "Двойной проход":
1.  **Проход 1:** Метод `_count_total_rows` читает все файлы целиком, чтобы посчитать количество строк для прогресс-бара.
2.  **Проход 2:** Метод `_convert_file` читает файлы заново для конвертации.

Для файла размером 1 ГБ это означает, что мы читаем 2 ГБ данных с диска. Это удваивает время ожидания перед стартом и общую нагрузку на диск.

**Решение:**
Изменить логику подсчета прогресса. Вместо подсчета *строк*, использовать размер файла в *байтах*.

**Алгоритм:**
1.  Получить общий размер всех входных файлов: `total_size_bytes = sum(os.path.getsize(f) for f in files)`.
2.  Во время чтения файла суммировать количество прочитанных байт: `processed_bytes += len(line.encode(encoding))`.
3.  Вычислять прогресс: `(processed_bytes / total_size_bytes) * 100`.

| Параметр | Описание |
|----------|----------|
| **Плюсы** | **Ускорение старта конвертации в 2 раза.** Конвертация начинается мгновенно. Снижение I/O нагрузки. |
| **Минусы** | Прогресс-бар может двигаться нелинейно (рывками), если строки имеют разную длину (некритично). |
| **Сложность** | Низкая (1-2 часа работы). |

## 2. Асинхронный анализ данных для UI (UX Optimization)

**Проблема:**
При добавлении большого файла или выборе столбца для фильтрации/разделения, приложение "фризится" (зависает). Это происходит потому, что методы `_update_column_combos` (чтение заголовков) и `_get_unique_values` (поиск уникальных значений) выполняются в **главном потоке (Main Thread)**.

**Решение:**
Вынести операции анализа (сканирование уникальных значений) в фоновые потоки с использованием `QRunnable` и `QThreadPool`.

**Алгоритм:**
1.  При выборе столбца в комбобоксе показывать индикатор загрузки (спиннер).
2.  Запускать `Worker`, который читает файл и собирает `set()` уникальных значений.
3.  По завершении `Worker` передает данные через сигнал в GUI и заполняет диалоговое окно.

| Параметр | Описание |
|----------|----------|
| **Плюсы** | **Интерфейс никогда не зависает.** Пользователь может продолжать работу, пока идет анализ. |
| **Минусы** | Усложнение кода GUI (нужно управлять состояниями "Загрузка", "Готово", "Ошибка"). |
| **Сложность** | Средняя. |

## 3. Оптимизация буферизации записи (IO Optimization)

**Проблема:**
При использовании `xlsxwriter` с `constant_memory=True` запись происходит построчно. Однако при разделении по столбцу (`split_column`) мы пишем в разные листы хаотично.
*XlsxWriter* в режиме константной памяти не любит переключения между листами: он может держать открытыми временные файлы для каждого листа. Если уникальных значений 1000, система упрется в лимит открытых файловых дескрипторов, либо производительность упадет из-за переключения контекста.

**Решение:**
Использовать промежуточную буферизацию (Batching) или сортировку.

**Вариант А (Сортировка - для CSV/TSV):**
Если позволяет место на диске, можно использовать утилиту (или алгоритм) внешней сортировки (External Sort) по столбцу разделения. Тогда все строки для "Лист 1" пойдут подряд, затем для "Лист 2". Это идеально для `xlsxwriter`.

**Вариант Б (Словари-буферы):**
Накапливать данные в памяти (например, по 1000 строк для каждого листа) и сбрасывать пачками.
*Примечание:* Сложно реализовать с `xlsxwriter` в режиме `constant_memory`, так как он ожидает последовательную запись.

**Решение "В лоб" (рекомендовано для текущей архитектуры):**
Если выбран режим "Разделения", сначала разбивать исходный TSV на множество временных маленьких TSV файлов (это очень быстро). Затем конвертировать каждый маленький TSV в отдельный лист Excel последовательно.

| Параметр | Описание |
|----------|----------|
| **Плюсы** | Стабильность при огромном количестве уникальных значений. Исключает `Too many open files`. |
| **Минусы** | Требуется временное место на диске. |
| **Сложность** | Высокая. |

## 4. Использование Pandas (Alternative Tech)

**Проблема:**
Python-циклы (`for row in reader`) работают медленно по сравнению с компилируемым кодом (C/C++).

**Решение:**
Использовать библиотеку **Pandas**.

**Пример:**
```python
import pandas as pd

# Чтение и конвертация в 3 строки кода
df = pd.read_csv(input_file, sep='	')
df.to_excel(output_path, engine='xlsxwriter')
```

**Нюансы с большими файлами:**
Pandas загружает все в RAM. Для больших файлов нужно использовать `chunksize`:
```python
with pd.ExcelWriter(output_path, engine='xlsxwriter') as writer:
    for chunk in pd.read_csv(input_file, sep='	', chunksize=100000):
        chunk.to_excel(writer, index=False)
```

| Параметр | Описание |
|----------|----------|
| **Плюсы** | **Ускорение обработки в 5-10 раз.** Значительное сокращение кода (меньше багов). Мощные возможности для Pivot Table "из коробки". |
| **Минусы** | **Размер приложения вырастет на ~50-100 МБ** (Pandas + NumPy тяжелые). Требует установки зависимостей. |
| **Сложность** | Низкая (переписывание кода), но высокая "цена" внедрения зависимости. |

## 5. Микро-оптимизации (Code Level)

1.  **Убрать проверки из цикла:**
    *   Сейчас внутри цикла `for row in reader` на каждой итерации проверяется `if self.stop_flag`, `if filter_idx is not None`.
    *   **Решение:** Создать разные функции-циклы для разных режимов (с фильтром и без). Проверка `stop_flag` каждые 1000 строк, а не каждую строку.

2.  **Кэширование атрибутов:**
    *   Вызов `row[idx]` быстрый, но если делать это миллионы раз, накладные расходы есть.
    *   В случае с CSV модулем это уже достаточно оптимизировано на уровне C.
