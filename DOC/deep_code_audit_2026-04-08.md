# Отчет о глубоком аудите кода TSV-Converter (2026-04-08)

## 1. 🧠 АНАЛИТИЧЕСКАЯ СПРАВКА (Общая оценка)

Архитектура проекта выстроена грамотно и профессионально. Приложение четко разделено на слои: бизнес-логику (`converter.py`), интерфейс (`gui.py`) и контроллер (`main.py`). Использование `PySide6` реализовано на высоком уровне с применением нативных стилей (отказ от QSS) и многопоточности (`QThread`, `QTimer`) для предотвращения блокировок главного потока (GUI).

Приложение устойчиво к ошибкам, использует `dataclasses` для строгой типизации конфигураций, а также оптимизировано для работы с большими файлами (лимит `xlsxwriter` в 1 млн строк, использование LRU-кэша для CSV-файлов, режим `constant_memory`).

Однако в ходе аудита были выявлены узкие места, связанные с потреблением оперативной памяти на сверхбольших массивах данных и потенциальными блокировками потока при фильтрации.

## 2. 🔍 РЕЗУЛЬТАТЫ АУДИТА ПО ФАЙЛАМ И "МЕРТВЫЙ КОД"

При анализе с помощью статических анализаторов (Ruff, Vulture) и глубокого ручного ревью было выявлено следующее:

### Статический анализ (Ruff & Vulture)
* **Синтаксис и стиль (Ruff):** Проблем не выявлено. Код полностью соответствует PEP8.
* **Мертвый код (Vulture):**
  * Были найдены подозрения на неиспользуемые методы в `gui.py` (`paintEvent`, `dragEnterEvent`, `dragMoveEvent`, `dropEvent`, `sizeHint`, `paint` и др.). **Это ложноположительные срабатывания (False Positives)** — данные методы являются переопределениями стандартных обработчиков событий Qt, они вызываются внутренним циклом событий (Event Loop).
  * `gui.py:757`: Переменная/сигнал `progress = Signal(int, str)` внутри `LoadingWorker`. **Действительно мертвый код.** Сигнал объявлен, но нигде не инициируется (`emit`) и не подключен к слотам в `main.py` или `gui.py`.

### Ручной аудит архитектуры
* `main.py`: Обработка событий `_update_total_rows`.
* `converter.py`: Алгоритмы агрегации сводных таблиц (`PivotTableProcessor`) и определение кодировки (`FileUtilities`).

## 3. ⚠️ КРИТИЧНЫЕ ПРОБЛЕМЫ И УЯЗВИМОСТИ

Хотя критических уязвимостей (RCE, SQLi, Path Traversal) в десктопном приложении без сети нет, присутствуют **проблемы производительности (DoS/OOM на локальной машине)**:

### 🔴 Угроза исчерпания памяти (OOM) в `PivotTableProcessor`
* **Локация:** `converter.py` -> `create_pivot_data`
* **Проблема:** Алгоритм собирает ВСЕ значения в списки в памяти: `pivot_data[row_key][col_key][key].append(value)`. Агрегация (сумма, среднее, макс/мин) происходит только после полного прочтения файла. Если файл содержит 5-10 миллионов строк и создается много групп, оперативная память будет переполнена списками из миллионов элементов, что приведет к аварийному завершению (Crash/OOM).
* **Сложность устранения:** Средняя.

### 🔴 Блокировка главного потока (GUI Freeze) при фильтрации
* **Локация:** `main.py` -> `_update_total_rows()`
* **Проблема:** Функция `_update_total_rows` вызывает `FileUtilities.count_rows`, которая читает файл целиком. Эта функция вызывается синхронно (в главном потоке) при любом изменении комбобокса фильтров (`_on_filter_column_selected`). На файлах размером более 1-2 ГБ выбор значения в фильтре вызовет зависание окна "Не отвечает" (Application Not Responding) на несколько секунд или минут.
* **Сложность устранения:** Средняя.

### 🟠 Потенциальные проблемы с чтением TSV в кодировке CP1251
* **Локация:** `converter.py` -> `FileUtilities.get_encoding`
* **Проблема:** Метод пытается прочитать файл как `utf-8`. Если падает `UnicodeDecodeError`, он откатывается на `windows-1251`, **НО только если файл имеет расширение `.csv`** (`if file_path.lower().endswith(".csv")`). Если загружен тяжелый файл `.tsv` в кодировке ANSI, скрипт попытается вернуть `utf-8` (упадет в общем `except`), что приведет к кракозябрам или ошибке при дальнейшей обработке.
* **Сложность устранения:** Низкая.

## 4. 🛠️ РЕКОМЕНДАЦИИ ПО УСТРАНЕНИЮ И ПОСЛЕДСТВИЯ

1. **Оптимизация сводных таблиц (Online Aggregation):**
   * Вместо `.append(value)` необходимо сразу накапливать сумму, количество и экстремумы.
   * *Как исправить:* В цикле чтения файла хранить словари с промежуточными результатами `{"sum": 0, "count": 0, "max": -inf, "min": inf}`. Это снизит потребление памяти при обработке сводных таблиц до константного уровня $O(1)$ относительно количества строк, зависящего только от числа уникальных групп.
   
2. **Асинхронный подсчет строк:**
   * *Как исправить:* Перенести логику `_update_total_rows` в отдельный `QThread` (например, использовать уже существующий `LoadingWorker`). Пока идет подсчет строк для выбранного фильтра, показывать пользователю анимацию загрузки или текст "Подсчет...". Это сделает UI идеально отзывчивым на любых объемах.

3. **Исправление функции `get_encoding`:**
   * *Как исправить:* Убрать жесткую привязку `endswith(".csv")` для fallback на `windows-1251` (или добавить `.tsv`, `.txt`).
   
4. **Очистка мертвого кода:**
   * *Как исправить:* Удалить неиспользуемый сигнал `progress = Signal(int, str)` из класса `LoadingWorker` в файле `gui.py`.
