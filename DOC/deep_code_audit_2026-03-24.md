## 2. Методика

Для аудита использовались несколько слоев проверки:

1. Полное ручное чтение каждого кодового файла.
2. Структурный обзор через Python AST для карты классов, методов и связей.
3. Проверки инструментами:
   - `python -m py_compile main.py gui.py converter.py`
   - `ruff check main.py gui.py converter.py`
   - `vulture main.py gui.py converter.py`
4. MCP analyzer применялся для Python-кода; для крупных файлов были дополнительно использованы локальные статические проверки, чтобы покрыть ограничения по времени/кодировке.
5. Context7 использовался для сверки ограничений `xlsxwriter`:
   - имя листа Excel: максимум 31 символ, есть ограничения на символы и апостроф в начале/конце;
   - лимит Excel-листа: 1,048,576 строк.

Примечание по `tree-hugger`: в репозитории не найдено ни одного файла `*.js`, `*.jsx`, `*.ts`, `*.tsx`, поэтому AST-анализатор JS/TS не имел целевых файлов.

## 3. Краткая архитектурная карта

### `main.py`

- Координационный слой между GUI и бизнес-логикой.
- Хранит `QSettings`, подключает сигналы окна и создает `TSVToExcelConverter`.
- Содержит логику завершения/ошибок конвертации и экспорт HTML-отчета.

### `gui.py`

- Самый крупный модуль.
- Содержит всю иерархию GUI: `MainWindow`, диалоги, кастомные виджеты, preview, настройки сводной таблицы, лог, loading overlay.
- Внутри есть как реальная логика UI, так и заглушки/остатки незавершенной логики.

### `converter.py`

- Утилиты работы с файлами.
- Поток конвертации TSV/CSV -> XLSX/CSV.
- Расчет прогресса.
- Построение и запись сводной таблицы.

### `TSV_Converter.spec`

- Конфигурация PyInstaller.
- Критичных ошибок в runtime-коде не содержит, но требует отдельного smoke-test сборки.

## 4. Сводка состояния

Общая оценка: код рабочий по базовому happy-path, но в проекте накопилось много архитектурных хвостов и несколько реально опасных дефектов.

Ключевые выводы:

- Есть дефекты, способные приводить к ложному статусу успешной конвертации.
- Есть дефекты, способные приводить к потере данных при split в CSV.
- Есть дефекты error-path'ов: приложение падает именно в ветках показа предупреждений.
- Есть заметный объем мертвого кода, неиспользуемых полей и недореализованных функций.
- Есть рассинхронизация между GUI-слоем и orchestration-слоем: часть кнопок подключена дважды.

## 5. Критические проблемы

### 5.1. Двойное подключение кнопок к слотам

Файлы и строки:

- `gui.py:2356`
- `gui.py:2474`
- `gui.py:2552`
- `gui.py:2557`
- `gui.py:2564`
- `main.py:67`
- `main.py:68`
- `main.py:69`
- `main.py:71`
- `main.py:72`

Суть:

- В `MainWindow` кнопки уже подключаются к внутренним методам окна.
- Затем `TSVConverterApp` повторно подключает те же `clicked`-сигналы к своим обработчикам.

Последствия:

- один клик вызывает две разные ветки логики;
- пользователь получает лишние логи и лишние побочные эффекты;
- для удаления это особенно опасно: сначала срабатывает GUI-заглушка, потом реальное удаление;
- поведение становится трудно предсказуемым и плохо тестируемым.

Симптомы в коде:

- `gui.py:2720` — заглушка предпросмотра только логирует сообщение;
- `gui.py:2819` — заглушка настроек сводной таблицы только логирует сообщение;
- `gui.py:2842` — заглушка открытия файла только логирует сообщение;
- `gui.py:2846` — заглушка удаления показывает диалог и пишет лог, но не удаляет файл;
- `gui.py:2858` — заглушка экспорта отчета только логирует сообщение.

Итог: сейчас GUI и orchestration-слой одновременно претендуют на владение одними и теми же кнопками.

### 5.2. Остановка конвертации трактуется как успешное завершение

Файлы и строки:

- `converter.py:452`
- `converter.py:453`
- `converter.py:474`
- `converter.py:478`
- `main.py:352`
- `main.py:372`

Суть:

- Если `stop_flag` поднят, конвертер пишет "Конвертация прервана пользователем" и делает `break`.
- После `break` код все равно идет в общую success-ветку, пишет "Конвертация завершена" и испускает `finished_signal`.
- GUI безусловно воспринимает `finished_signal` как успешный финал и показывает сообщение "Конвертация завершена успешно!".

Последствия:

- пользователь видит ложный green status;
- можно открыть/удалить частичный результат как будто он полный;
- трудно понять, была ли остановка штатной, частичной или аварийной.

Это один из самых опасных дефектов во всем проекте, потому что он искажает итог операции.

### 5.3. Тихая потеря данных при split в CSV из-за коллизий имен файлов

Файлы и строки:

- `converter.py:292`
- `converter.py:320`
- `converter.py:685`
- `converter.py:693`
- `converter.py:694`
- `converter.py:695`

Суть:

- Для split-CSV имя файла строится через `FileUtilities.sanitize_sheet_name()`.
- Эта функция предназначена под ограничения Excel-листов, а не под файловую систему.
- В `_process_split_row()` она вызывается без общего `used_names`.
- Разные исходные значения могут быть преобразованы в один и тот же `safe_key`.

Примеры коллизий:

- `A/B` и `A:B` -> оба превращаются в `A_B`;
- длинные ключи, отличающиеся после 29-го символа, обрезаются одинаково;
- два разных ключа открывают один и тот же файл с режимом `"w"`, то есть более поздний ключ перезапишет ранний.

Последствия:

- silent overwrite;
- потеря части выгрузки без исключения и без предупреждения;
- пользователь получает формально корректный набор CSV, но с неправильными данными.

### 5.4. После split-CSV приложение считает выходом каталог, но удаляет его как файл

Файлы и строки:

- `converter.py:683`
- `main.py:469`
- `main.py:482`

Суть:

- После split-CSV `self.output_file_path` выставляется в `self.output_directory`, то есть в папку.
- Кнопка "Удалить файл" далее делает `os.remove(path)` без проверки, что это именно файл.

Последствия:

- при split-режиме удаление ломается исключением;
- UI семантически вводит пользователя в заблуждение: он работает не с файлом, а с директорией;
- если исключение не перехвачено в Qt-слоте, пользователь видит аварийный сценарий вместо штатного сообщения.

### 5.6. Sanitizer имен листов не покрывает все ограничения `xlsxwriter`

Файлы и строки:

- `converter.py:292`
- `converter.py:307`
- `converter.py:916`
- `converter.py:940`
- `converter.py:954`
- `converter.py:1032`
- `converter.py:1072`
- `converter.py:1102`

Проверка через Context7 (`xlsxwriter`) подтверждает:

- имя листа не должно быть длиннее 31 символа;
- запрещены символы `[ ] : * ? / \`;
- имя не должно начинаться или заканчиваться апострофом.

Проблема:

- текущая функция не обрабатывает апостроф в начале/конце;
- следовательно, некоторые значения split-столбца могут вызвать runtime-ошибку при `workbook.add_worksheet(...)`.

Последствия:

- конвертация падает на реальных пользовательских данных;
- ошибка всплывает только на определенных значениях и потому плохо воспроизводима.

## 6. Высокие проблемы

### 6.1. Прогресс, ETA и счетчик строк считаются некорректно для нескольких файлов и фильтрации

Файлы и строки:

- `converter.py:447`
- `converter.py:904`
- `converter.py:981`
- `converter.py:1000`
- `converter.py:1025`
- `converter.py:1059`
- `converter.py:1093`

Суть:

- `self.total_rows` считается по всем входным файлам суммарно.
- Но `progress_tracker.update(row_num)` получает `row_num`, который начинается заново с `1` для каждого отдельного файла.
- При активном фильтре `row_num` увеличивается по прочитанным строкам, а `total_rows` уже учитывает только строки, прошедшие фильтр.

Последствия:

- прогресс по нескольким файлам может идти назад или замирать;
- ETA и скорость становятся ложными;
- детализация `processed_rows / total_rows` вводит пользователя в заблуждение.

Дополнительный дефект:

- `self.processed_rows` в `TSVToExcelConverter` объявлен (`converter.py:423`), но фактически не участвует в расчете.

### 6.2. Предпросмотр файла несовместим с реальной логикой чтения

Файлы и строки:

## 8. Низкие проблемы и архитектурные шероховатости

### 8.1. Лимит строк на лист занижен относительно реального лимита Excel

Файлы и строки:

- `converter.py:903`
- `converter.py:1024`

Сейчас в коде используется `MAX_ROWS = 1000000`, что безопасно, но приводит к преждевременному созданию дополнительных листов.

### 8.2. Sanitizer режет имена до 29 символов, хотя Excel допускает 31

Файлы и строки:

- `converter.py:312`

Это не ломает функциональность напрямую, но unnecessarily уменьшает информативность имен листов и имен split-CSV файлов.

### 8.3. `TSV_Converter.spec`: исключение `pillow` может быть неэффективным

Файлы и строки:

## 13. Итоговый вердикт

Проект не выглядит заброшенным: базовая функциональность конвертации и значительная часть UI уже реализованы. Но кодовая база находится в состоянии, когда несколько seemingly мелких компромиссов начали складываться в системные проблемы:

- orchestration и GUI пересекаются по ответственности;
- error-path'ы хуже, чем happy-path;
- часть настроек и элементов UI только имитирует функциональность;
- есть реальные сценарии тихой потери данных.

Если кратко: приложение уже полезно, но его нельзя считать надежным без исправления проблем из разделов 5 и 6.
